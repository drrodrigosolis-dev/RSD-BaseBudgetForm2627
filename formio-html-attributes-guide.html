Form.io HTML Attributes Quick Reference (Comprehensive)

This guide covers what you can personalize using the HTML Attributes tab (and component class hooks) when you do not have full app-level stylesheet access. It focuses on class/style/id/data/aria attributes, practical limits, and concrete recipes for all major component families.

1) What This Approach Can Do

In Form.io, HTML Attributes are usually key/value pairs applied to rendered elements or wrappers. This gives strong control over local presentation and metadata, even when global CSS editing is restricted.

Strong use cases: local spacing, borders, backgrounds, width behavior, per-component card styling, custom IDs, test hooks, accessibility attributes, and integration tags for scripts.
Limits: inline styles cannot target pseudo-classes (:hover, :focus), sibling relationships, or deeply nested descendants unless you can attach classes and style them elsewhere.
Best practice: use HTML Attributes to attach predictable classes/ids/data hooks, then style those hooks using the CSS-injection method whenever possible for maintainability.

3) Atomic Attribute Reference (A-Z Dictionary)

This section is strictly disaggregated and alphabetized. Use it like a dictionary: one attribute/property at a time.

3.1 How To Write HTML Attributes Correctly

<!-- One attribute -->
class="fio-card"

<!-- Multiple attributes -->
class="fio-card"
id="employmentSection"
data-test="employment-section"
aria-label="Employment Section"

<!-- style attribute syntax: property:value; property:value; -->
style="background:#f8fbff; border:1px solid #cad6ef; border-radius:10px;"

<!-- Important: include semicolons between properties -->
style="display:flex; gap:8px; align-items:center;"

6) Advanced Patterns (IDs, Data Hooks, ARIA)

6.1 Stable IDs For Anchors And Integrations

Use this for deep links and scripted scroll/focus behavior.

6.2 Data Attributes For QA Automation

data-test="employee-name"

These are ideal for Playwright/Cypress selectors because they are stable compared with generated class names.

7) Hard Limits + Workarounds

Need Why HTML Attributes Alone Are Limited Workaround
Hover/focus visual effects Inline style cannot define pseudo-classes. Add class hook and define :hover/:focus in injected CSS.
Consistent theme across whole form Repeating inline style is hard to maintain. Use one wrapper class + centralized CSS tokens.
Styling child internals of complex widgets Attributes may only apply at outer wrapper. Use wrapper class and descendant selectors in CSS approach.
Responsive behavior Inline styles do not support media queries. Assign classes now; implement breakpoints in CSS-injection file.
Validation state theming State classes are runtime-driven and need selectors. Style .has-error/.is-invalid in scoped CSS.
Practical rule: use HTML Attributes for structure, identity, and local style; use CSS injection for reusable theme logic.

8) QA Checklist

  1. Every key section has a stable id or data-test hook.
  2. Wrapper classes follow a naming convention (example: fio-*).
  3. Inline styles are short and purposeful, not repeated excessively.
  4. No invalid HTML/CSS values in style attributes.
  5. ARIA attributes are meaningful and not contradictory.
  6. Mobile width behavior checked for local grid/flex styles.
  7. If combined with CSS injection, selectors target the assigned classes/ids exactly.

Keyboard tip: when validating focus behavior in the final app, use Tab and Shift + Tab to ensure your attribute-based structure remains usable.
